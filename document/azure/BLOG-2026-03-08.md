# Azure AKS 멀티클러스터 Addon 설치 실전 장애 분석 — Cilium CNI 재시작이 연쇄 장애를 유발한 이유

**날짜**: 2026-03-08
**환경**: AKS 1.35 / Azure CNI Overlay + Managed Cilium / Korea Central

---

## 요약

OpenTofu로 AKS 3개 클러스터(mgmt/app1/app2) 배포 후 약 20개 addon을 자동 설치하는 과정에서, 특정 addon이 3시간 이상 `ContainerCreating` 상태로 멈추는 장애가 발생했습니다. 처음에는 노드 리소스 부족을 의심했지만, 실제 원인은 **Cilium CNI agent 재시작 중에 다른 addon Pod들이 IP를 요청**한 것이었습니다.

---

## 증상

```
ama-logs DaemonSet: ContainerCreating (77분 째 stuck)
defender-collector: Pending (92분 째)
otel-collector: ContainerCreating (50분 째)
kyverno-admission-controller: CreateContainerError (반복)
```

`kubectl describe pod` 결과:

```
Warning  FailedCreatePodSandBox
  Failed to create pod sandbox: plugin type="cilium-cni" failed (add):
  unable to connect to Cilium agent: Cilium API client timeout exceeded

Warning  FailedCreatePodSandBox
  http://localhost:10090/network/requestipconfigs: connection refused
```

---

## 원인 분석

### 1. Cilium agent가 죽어있었다

Azure CNI Overlay + Managed Cilium 환경에서 **ACNS (Advanced Container Networking Services)를 활성화하면 Cilium agent DaemonSet이 재시작**됩니다.

```bash
# 이 명령이 Cilium을 재시작시킨다
az aks update --enable-acns --resource-group rg-k8s-app1 --name aks-app1
```

Azure CLI는 AKS control plane 업데이트가 완료되면 반환되지만, **노드 위의 Cilium DaemonSet 롤아웃은 별도로 진행**됩니다. Cilium이 재시작 중인 동안 새 Pod가 생성되면:

- CNI plugin이 Cilium agent에 IP를 요청 (`localhost:10090`)
- Cilium agent가 아직 준비 안 됨 → `connection refused`
- Pod sandbox 생성 실패 → `ContainerCreating` 무한 대기

### 2. 설치 순서가 잘못됐다

```
# 기존 순서 (문제 있음)
00-priority-classes → 01-cert-manager → ... → 13-hubble → 16-otel-collector
#                                              ↑ ACNS 활성화         ↑ CNI 재시작 중에 설치 시도
```

`13-hubble.sh`가 중간 순서라 ACNS 활성화 후 Cilium이 재시작되는 동안 `16-otel-collector.sh`가 실행되어 ContainerCreating 상태로 멈췄습니다.

### 3. Kyverno containerd zombie 충돌

CNI 실패로 컨테이너가 완전히 종료되지 않은 채 containerd에 잔류했습니다. 이후 Kyverno가 재시작을 시도할 때 동일한 컨테이너 이름을 예약하려 하면:

```
Error: failed to reserve container name
  "kyverno_kyverno-admission-controller-..._1":
  name "..." is reserved for "86503e0561757690b4..."
```

이 오류는 `CreateContainerError`로 표시되며 수동 개입 없이는 자동 복구가 안 됩니다.

### 4. reg.kyverno.io 이미지 pull 극히 느림

Kyverno 기본 이미지 레지스트리 `reg.kyverno.io`에서 이미지를 pull하는 데 30분 이상 소요됐습니다. 이 긴 pull 시간이 CNI 장애와 겹쳐 더 오래 stuck 상태를 유지시켰습니다.

## 해결 방법

### Fix 1: `13-hubble.sh` — Cilium DaemonSet 준비 대기 추가

```bash
# ACNS 활성화
az aks update --resource-group "${RG}" --name "${CLUSTER_NAME}" --enable-acns

# Cilium DaemonSet 완전 롤아웃 대기
kubectl rollout status daemonset/cilium -n kube-system --timeout=10m

# CNS(Container Network Service)가 모든 노드에서 준비될 때까지 추가 대기
sleep 30
```

### Fix 2: `install.sh` — Hubble을 맨 먼저 실행

```
# 변경 후 순서
13-hubble (ACNS + Cilium 안정화) → 00-priority-classes → 01-cert-manager → ...
```

ACNS가 가장 먼저 완료되고 Cilium이 안정화된 뒤에 나머지 addon을 설치합니다.

### Fix 3: `04-istio.sh` — istiod 준비 대기 추가

```bash
az aks mesh enable --revision asm-1-28 ...

# istiod deployment가 완전히 Ready될 때까지 대기
# (이후 04b-istio-mtls.sh에서 webhook 오류 방지)
kubectl rollout status deployment/istiod-asm-1-28 -n aks-istio-system --timeout=10m
```

### Fix 4: `05-kyverno.sh` — zombie Pod 정리 + 이미지 레지스트리 변경

```bash
# 재설치 전 Terminating / CreateContainerError Pod 강제 제거
kubectl get pods -n kyverno --no-headers \
  | awk '$3=="Terminating" || $3=="CreateContainerError" {print $1}' \
  | xargs -r kubectl delete pod -n kyverno --force --grace-period=0

# 이미지 레지스트리 reg.kyverno.io → ghcr.io (pull 속도 개선)
helm upgrade --install kyverno kyverno/kyverno \
  --set global.image.registry=ghcr.io \
  ...
```

## 얻은 교훈

### 1. AKS addon 순서는 CNI 생명주기를 기준으로 설계해야 한다

`az aks update`는 control plane 업데이트 완료 시점을 반환하지만, **DaemonSet 롤아웃은 별도로 진행**됩니다. Cilium처럼 모든 Pod에 영향을 주는 컴포넌트는 반드시 `kubectl rollout status`로 추가 확인해야 합니다.

### 2. 외부 레지스트리 의존성을 파악하라

`reg.kyverno.io`, `docker.io` 등 외부 레지스트리에서 이미지를 pull하면 한국 리전에서 속도가 매우 느릴 수 있습니다. `ghcr.io` 또는 ACR 미러링을 활용하세요.

### 3. containerd zombie는 강제 삭제로 해결

`CreateContainerError`가 반복된다면 `kubectl delete pod --force --grace-period=0`으로 Pod를 강제 삭제해 containerd 상태를 초기화해야 합니다. `kubectl delete pod` (grace period 기본값)로는 해결이 안 됩니다.

### 4. 멱등성은 "현재 상태 확인" 후 결정해야 한다

Hubble ACNS처럼 이미 활성화된 경우 Cilium을 재시작시키면 안 됩니다. 모든 설치 스크립트에 멱등성 체크를 추가하세요:

```bash
ACNS_ENABLED=$(az aks show ... --query 'networkProfile.advancedNetworking.observability.enabled' -o tsv)
if [[ "${ACNS_ENABLED}" == "true" ]]; then
  echo "Already enabled — skip"
  exit 0
fi
```
